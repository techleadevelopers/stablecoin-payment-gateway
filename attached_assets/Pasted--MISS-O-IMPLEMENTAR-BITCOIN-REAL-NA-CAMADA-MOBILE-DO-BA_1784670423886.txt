# MISSÃO — IMPLEMENTAR BITCOIN REAL NA CAMADA MOBILE DO BACKEND

Atue como um **Staff Backend Engineer especializado em Bitcoin, segurança financeira, sistemas custodiais e infraestrutura de pagamentos**.

Implemente suporte real a Bitcoin no backend utilizado pelo aplicativo mobile.

A implementação deve ser de nível produção, com:

* endereços Bitcoin reais;
* carteira HD;
* leitura de UTXOs;
* saldo confirmado e pendente;
* detecção de depósitos;
* criação de transações;
* cálculo de fee;
* assinatura;
* broadcast;
* acompanhamento de confirmações;
* idempotência;
* auditoria;
* testes automatizados.

## REGRA PRINCIPAL

Não altere, não misture e não quebre os fluxos existentes de:

* BSC;
* Polygon;
* EVM;
* USDT;
* USDC;
* buy;
* sell;
* swap;
* settlement;
* Agent Rail;
* NFC;
* signer EVM;
* workers existentes.

Bitcoin deve ser implementado como uma **rail independente**.

Não tente reaproveitar lógica de:

* nonce EVM;
* gas;
* contratos Solidity;
* endereços `0x`;
* signer EVM;
* RPC EVM;
* transferências ERC-20.

Bitcoin utiliza:

* endereços Bitcoin;
* UTXOs;
* inputs e outputs;
* sats;
* fees em sats/vbyte;
* transações Bitcoin;
* confirmações de bloco.

## ESCOPO DESTA IMPLEMENTAÇÃO

O foco é exclusivamente a camada de backend consumida pelo aplicativo mobile.

Não redesenhe o sistema inteiro.

Não altere o funcionamento atual dos ativos EVM.

Implemente a infraestrutura BTC de modo modular e isolado, pronta para ser ativada gradualmente.

## FASES OBRIGATÓRIAS

Implemente nesta ordem:

1. infraestrutura e configuração BTC;
2. geração de endereço de recebimento;
3. detecção de depósitos;
4. saldo confirmado e pendente;
5. envio de BTC;
6. worker de confirmações;
7. integração com os endpoints mobile;
8. testes unitários, integração e adversariais;
9. documentação operacional.

Cada fase deve funcionar antes de avançar para a seguinte.

# 4. MODELO DE CARTEIRA HD

Implemente uma estratégia HD adequada para Bitcoin.

Preferência:

* BIP32;
* BIP39 quando houver seed;
* BIP84 para Native SegWit;
* endereços `bc1q...` em mainnet;
* endereços correspondentes à rede em testnet/regtest.

Caminho conceitual:

```text
m/84'/0'/0'/0/index
```

Para testnet:

```text
m/84'/1'/0'/0/index
```

## Segurança

A API pública do backend deve gerar endereços usando `xpub`, sem precisar carregar chaves privadas.

A assinatura deve ficar isolada.

Quando possível:

* address service usa somente `xpub`;
* signer usa seed/xpriv criptografada;
* signer não expõe endpoint genérico de assinatura;
* nenhuma função de assinatura arbitrária;
* nenhuma função equivalente a `execute arbitrary transaction`.

Não derive novamente um índice já usado para outro usuário.

A alocação de índice deve ser transacional e possuir restrição única.

Crie ou adapte um endpoint mobile para receber BTC.

## Comportamento

* Se o usuário não possui endereço, gere um de forma transacional.
* Se já possui, retorne o endereço existente.
* Requisições concorrentes não podem gerar vários endereços acidentalmente.
* O endpoint deve ser autenticado.
* Registre evento de auditoria sem registrar segredo.
* Valide que o endereço corresponde à rede configurada.

O endereço BTC deve ser armazenado separadamente do `wallet_address` EVM.

# 7. DETECÇÃO DE DEPÓSITOS

Implemente worker de depósitos BTC.

Responsabilidades:

1. obter endereços ativos;
2. consultar UTXOs;
3. inserir novos UTXOs de modo idempotente;
4. identificar depósitos pendentes;
5. atualizar confirmações;
6. mover depósito para confirmado ao atingir o mínimo;
7. detectar reorganização de blockchain;
8. marcar UTXOs que desapareceram ou ficaram órfãos;
9. publicar eventos internos após commit.

Eventos sugeridos:

```text
btc.deposit.detected
btc.deposit.confirmed
btc.deposit.reorged
```

Payload mínimo:

```json
{
  "user_id": "...",
  "network": "bitcoin",
  "txid": "...",
  "vout": 0,
  "amount_sats": 150000,
  "confirmations": 3,
  "address": "..."
}
```

## Regras

* Nunca creditar o mesmo `txid:vout` duas vezes.
* Saldo pendente e confirmado devem ser separados.
* Depósitos não confirmados não podem ser usados como saldo disponível.
* Reprocessar um bloco deve ser seguro.
* Falha temporária do provider não deve apagar dados previamente válidos.
* Use retry com backoff e jitter.
* Evite scans concorrentes do mesmo endereço.
* Implemente lock distribuído ou liderança de worker quando houver múltiplas instâncias.

# 8. SALDO BTC

Crie endpoint:

```http
GET /api/mobile/wallets/btc/balance
```

Resposta:

```json
{
  "asset": "BTC",
  "network": "bitcoin",
  "confirmed_sats": 250000,
  "pending_sats": 50000,
  "reserved_sats": 20000,
  "available_sats": 230000,
  "confirmed_btc": "0.00250000",
  "pending_btc": "0.00050000",
  "available_btc": "0.00230000",
  "minimum_confirmations": 3,
  "updated_at": "..."
}
```

## Fórmulas

```text
confirmed = soma dos UTXOs confirmados e não gastos
pending = soma dos UTXOs ainda sem confirmações suficientes
reserved = soma dos UTXOs reservados para withdrawals
available = confirmed - reserved
```

Use `int64` para sats.

Nunca use `float32` ou `float64` para representar BTC financeiro.

Para respostas em BTC:

* use decimal exato;
* sempre 8 casas decimais quando apropriado;
* conversão determinística entre BTC e sats.

```text
1 BTC = 100000000 sats
```

## Processo obrigatório

O envio deve executar:

1. autenticação;
2. validação do ativo e rede;
3. validação do endereço;
4. validação do valor;
5. validação dos limites;
6. validação da idempotência;
7. cálculo do request hash;
8. leitura de saldo disponível;
9. estimativa segura da fee;
10. seleção de UTXOs;
11. reserva atômica dos UTXOs;
12. criação do output de destino;
13. criação do output de change quando necessário;
14. validação de dust;
15. montagem da PSBT ou transação;
16. assinatura isolada;
17. validação da transação assinada;
18. broadcast;
19. persistência do txid;
20. atualização dos estados dos UTXOs;
21. publicação de evento após commit.

## UTXO locking

Duas requisições simultâneas nunca podem gastar o mesmo UTXO.

Utilize:

* transação PostgreSQL;
* `SELECT ... FOR UPDATE SKIP LOCKED`, quando adequado;
* status `reserved`;
* expiration para reservas abandonadas;
* unique constraints;
* idempotência.

Não mantenha transação de banco aberta durante chamada longa ao provider.

Use uma máquina de estados segura:

```text
created
→ utxos_reserved
→ transaction_built
→ signed
→ broadcast
→ confirmed
```

Em falha recuperável:

* preservar dados suficientes para reconciliação;
* não liberar UTXOs indiscriminadamente depois que a transação tiver sido assinada ou enviada;
* consultar o provider antes de concluir que o broadcast falhou.

## Validação de endereço

Valide:

* checksum;
* formato;
* rede;
* tipo de script suportado;
* tamanho;
* caracteres;
* endereços mainnet em testnet;
* endereços testnet em mainnet.

Suporte inicialmente:

* P2WPKH Native SegWit;
* Bech32.

Outros formatos só devem ser aceitos quando a biblioteca usada os validar corretamente.

## Fees

Implemente políticas:

```text
fast
normal
economy
```

Mapeadas para número-alvo de blocos.

A estimativa deve utilizar sats/vbyte.

Regras:

* aplicar mínimo configurado;
* aplicar máximo configurado;
* rejeitar estimativa fora do limite;
* possuir fallback configurável;
* registrar a origem da fee;
* nunca usar uma fee fixa silenciosamente;
* calcular fee final com base no tamanho virtual da transação.

## Change

O change deve ser enviado para endereço controlado pela carteira Bitcoin.

Não envie change para:

* endereço do destinatário;
* endereço EVM;
* endereço arbitrário do request.

O change address deve ser derivado e registrado com segurança.

Evite criar output de change abaixo do dust limit.

Quando o change for dust, avalie adicioná-lo à fee conforme política explícita.

A camada de assinatura deve:

* receber apenas transações Bitcoin previamente validadas;
* conferir network;
* conferir inputs;
* conferir outputs;
* conferir valor;
* conferir fee;
* conferir fee rate;
* conferir change;
* conferir ownership dos derivation paths;
* recusar outputs adicionais inesperados;
* recusar fee acima do máximo;
* recusar destino alterado;
* recusar assinatura duplicada indevida.

A seed ou xpriv deve permanecer:

* criptografada em repouso;
* fora de logs;
* fora de respostas HTTP;
* fora de mensagens de erro;
* fora de telemetry;
* fora do banco em texto puro.

Não crie endpoint administrativo genérico que permita assinar qualquer PSBT arbitrária.

# 11. BROADCAST E RECONCILIAÇÃO

Após assinar:

1. calcule o txid localmente;
2. envie a raw transaction;
3. compare o txid local com o retorno do provider;
4. registre a tentativa;
5. trate timeout como estado desconhecido, não como falha definitiva;
6. consulte o txid antes de tentar rebroadcast;
7. permita rebroadcast idempotente da mesma raw transaction;
8. nunca construa uma nova transação automaticamente usando os mesmos UTXOs sem reconciliação.

Eventos:

```text
btc.withdrawal.created
btc.withdrawal.signed
btc.withdrawal.broadcast
btc.withdrawal.confirmed
btc.withdrawal.failed
btc.withdrawal.dropped
```
