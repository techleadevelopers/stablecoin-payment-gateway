O próximo passo estratégico para garantir que nenhuma alteração futura quebre essas travas de segurança é criar um Suite Unificado de Testes de Estresse e Simulação de Falhas Locais. Em vez de rodar testes isolados em Go, vamos construir scripts robustos que sobem o ecossistema híbrido, simulam a destruição de infraestrutura em tempo real (queda de banco, lag em RPC da BSC/Polygon, perda de mTLS da Efí) e medem as métricas de SLO.

Aqui está o arsenal técnico necessário para consolidar esse avanço:

🛠️ 1. O Script de Caos e Estresse Local (tests/chaos_suite.sh)
Este script em Shell orquestra um teste de estresse real usando o k6 que você já configurou, mas injeta caos de infraestrutura no meio do disparo para avaliar se a resiliência do Paymaster (Idempotência, Backoff com Jitter, DLQ) e do Human Rail funciona sob fogo cruzado.

Crie o arquivo tests/chaos_suite.sh:

Bash
#!/usr/bin/env bash

set -euo pipefail

# Configurações de Cores para o terminal
GREEN='\033[0;32m'
RED='\033[0;31m'
YELLOW='\033[1;33m'
NC='\033[0m'

echo -e "${YELLOW}[1/4] Inicializando ambiente de simulação adversária local...${NC}"

# Garante que o ambiente de teste local está ativo (Postgres + Redis)
docker compose -f docker-compose.test.yml up -d --build

# Aguarda os bancos estarem prontos
echo "Aguardando prontidão do banco de dados..."
sleep 5

echo -e "${YELLOW}[2/4] Disparando k6 Spike Test contra o Paymaster e MCP...${NC}"
# Roda o k6 em background para podermos injetar o caos
k6 run tests/paymaster_stress.js \
  -e BASE_URL=http://localhost:8080 \
  -e API_KEY_LIVE=sk_live_test_chaos_key_123 \
  -e API_KEY_TEST=sk_test_test_chaos_key_123 &
K6_PID=$!

sleep 5

# --- INJEÇÃO DE CAOS 1: Queda intermitente do cache de Idempotência (Redis) ---
echo -e "${RED}🔥 [CAOS] Derrubando o Redis no meio da enxurrada para testar fallback de memória...${NC}"
docker compose -f docker-compose.test.yml stop redis

sleep 8

echo -e "${GREEN}🔄 [RESTAURAÇÃO] Subindo o Redis novamente...${NC}"
docker compose -f docker-compose.test.yml start redis

sleep 5

# --- INJEÇÃO DE CAOS 2: Latência artificial na camada de Banco (Postgres) ---
echo -e "${RED}🔥 [CAOS] Introduzindo 500ms de latência na interface de rede do Postgres (Simulação de gargalo)...${NC}"
# Se o tc (traffic control) estiver disponível no container:
docker exec chainfx-postgres-test apt-get update && docker exec chainfx-postgres-test apt-get install -y iproute2 || true
docker exec chainfx-postgres-test tc qdisc add dev eth0 root netem delay 500ms || true

sleep 10

echo -e "${GREEN}🔄 [RESTAURAÇÃO] Removendo latência do Postgres...${NC}"
docker exec chainfx-postgres-test tc qdisc del dev eth0 root || true

# Aguarda a finalização do k6
echo -e "${YELLOW}[3/4] Aguardando conclusão do motor k6...${NC}"
wait $K6_PID

echo -e "${GREEN}[4/4] Caos concluído. Verificando logs por vazamento de erros 5xx ou duplicações...${NC}"

# Verifica se houve panics ou vazamento de estado nos logs do container Go
if docker logs chainfx-api-test 2>&1 | grep -iq "panic"; then
    echo -e "${RED}🚨 FALHA GRAVE DETECTADA: O sistema sofreu um Panic sob estresse/caos!${NC}"
    exit 1
else
    echo -e "${GREEN}🎉 SUCESSO: O sistema se degradou de forma limpa retornando 429/409/503 e não crashou!${NC}"
fi
🚦 2. Adicionando SLOs Híbridos ao tests/paymaster_stress.js
Para que o script de caos acima valide as camadas de forma unificada, precisamos adicionar métricas focadas nos SLOs do Human Rail (Notificações Pix Efí) e no Bypass de Assinatura.

Atualize os thresholds e adicione o cenário do Human Rail dentro do seu arquivo tests/paymaster_stress.js:

JavaScript
import http from 'k6/http';
import { check, sleep } from 'k6';
import { Trend, Counter } from 'k6/metrics';

// Métricas customizadas para auditoria local de SLO
const humanRailSettlementDuration = new Trend('human_rail_settlement_duration');
const hmacValidationFailures = new Counter('hmac_validation_failures');

export const options = {
    scenarios: {
        // ... Seus 5 cenários anteriores (paymaster_spike, idempotency_collision, etc.) ...
        
        // CENÁRIO 6 (Novo): Simulação de enxurrada de Webhooks Pix (Human Rail)
        human_rail_flood: {
            executor: 'constant-arrival-rate',
            rate: 50, // 50 notificações por segundo da Efí
            timeUnit: '1s',
            duration: '20s',
            preAllocatedVUs: 10,
            maxVUs: 50,
        },
    },
    thresholds: {
        http_req_duration: ['p95<400'], // Média geral de resposta
        human_rail_settlement_duration: ['p95<300'], // Resolução de concorrência Pix tem que ser ultra rápida
        infra_errors: ['rate<0.01'], // Menos de 1% de erros internos (5xx), mesmo sob caos
    },
};

export default function () {
    const baseUrl = __ENV.BASE_URL || 'http://localhost:8080';
    
    // Simulação do disparo contra o Webhook da Efí
    const efiUrl = `${baseUrl}/v1/webhooks/efi`;
    const efiPayload = JSON.stringify({
        event: 'pix_received',
        txid: 'tx_chaos_test_987654321',
        value: 25050 // R$ 250,50
    });

    const efiParams = {
        headers: {
            'Content-Type': 'application/json',
            'X-Efi-Signature': 'valid_hmac_signature_computed_locally'
        },
    };

    // Dispara contra o Human Rail
    const startTime = Date.now();
    const resEfi = http.post(efiUrl, efiPayload, efiParams);
    
    if (resEfi.status === 200 || resEfi.status === 409) {
        humanRailSettlementDuration.add(Date.now() - startTime);
    } else if (resEfi.status === 401) {
        hmacValidationFailures.add(1);
    }

    sleep(0.1);
}
🔒 3. Script de Validação de Segurança Estática (Pre-Commit / CI Gate)
Ataques adversários também ocorrem se dependências externas forem atualizadas com códigos maliciosos. Como você usa criptografia pesada, gerenciamento de chaves via signer service externo e parsers customizados da EVM, precisamos de um script de linting de segurança para rodar localmente antes de empurrar para produção.

Crie o arquivo scripts/security_lint.sh:

Bash
#!/usr/bin/env bash
set -euo pipefail

echo "🔍 Executando Auditoria de Vulnerabilidade nas Dependências Go (govulncheck)..."
# Instala e executa a ferramenta de análise de vulnerabilidade oficial da equipe do Go
go install golang.org/x/vuln/cmd/govulncheck@latest
govulncheck ./...

echo "🎯 Executando Verificação Estática de Segurança (Gosec)..."
# Instala o Gosec se não estiver presente e analisa brechas como concorrência errada ou pseudo-randomização
if ! command -v gosec &> /dev/null; then
    go install github.com/securego/gosec/v2/cmd/gosec@latest
fi
gosec -exclude=G104,G404 ./...

echo "✅ Verificações estáticas concluídas sem falhas críticas de segurança de código!"
